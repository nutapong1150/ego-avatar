# เนตรกลยุทธ์ SaaS (SaaS Strategy Lens)

> **แหล่งที่มา:** 5 คลิป Torpenguin × ดร.เก้ (ดร.ปรเมษฐ์ ชุ่มยิ้ม, FoodInnopolis)
> ปรับบริบทจากธุรกิจอาหาร → **สร้างและขายระบบ SaaS**
> เสริมด้วย Best Practice จาก Ash Maurya, Dave McClure, Patrick Campbell
>
> **ปรัชญาแก่น:** "สิ่งที่คุณขายไม่สำคัญ — วิธีที่คุณขายต่างหากที่สำคัญ"

## เมื่อไรจะใช้ Skill นี้

- ท่านต้องการ **สร้าง SaaS ใหม่** หรือ **คิดไอเดีย SaaS**
- ท่านถามว่า "จะขายให้ใคร" "ใครจะยอมจ่าย" "ทำไมลูกค้าจะเปลี่ยนมาใช้เรา"
- ท่านต้องการ **วิเคราะห์คู่แข่ง** หรือหา **Positioning** ของ SaaS
- ท่านต้องการ **ตั้งราคา** หรือออกแบบ **Pricing Model**
- ท่านพูดคำว่า: BMC, Lean Canvas, JTBD, Design Thinking, Product-Market Fit, Switch Moment, Four Forces, AARRR, Value Metric

---

## 🔗 Pipeline ครบวงจร: ใช้ 7 Frameworks ร่วมกัน

```
[1] Design Thinking — Empathize
      + Switch Interview (5 คำถาม)
            ↓ ค้นพบปัญหา + Four Forces จริงจากลูกค้า
[2] Lean Canvas — Problem + Customer + UVP
      + AARRR Key Metrics
            ↓ วาง Business Model + ตั้งเป้า Metrics
[3] JTBD + Four Forces
            ↓ ออกแบบ Feature + Messaging ที่ตรงจุด
[4] Design Thinking — Ideate + Prototype + Test
            ↓ สร้างและทดสอบ MVP
[5] BMC (9 ช่อง)
      + SaaS Pricing Strategy
            ↓ วาง Revenue Model + Scale
[6] AARRR Funnel Tracking
            ↓ วัดผล + ปรับจูน
[7] Product-Market Fit → Business-Market Fit
            ↓ 🚀 SaaS ที่ขายได้จริง
```

## Framework 1: Business Model Canvas (BMC) — แผนที่ธุรกิจ 9 ช่อง

> 📖 สกัดจาก: คลิป "วางกลยุทธ์ธุรกิจอาหาร ด้วยกระดาษแผ่นเดียว" (J3RJhZQ3Sns)

### 9 ช่องที่ต้องกรอกสำหรับ SaaS:

| ช่อง BMC | คำถามสำคัญสำหรับ SaaS |
|---|---|
| **Customer Segments** | กลุ่มเป้าหมาย: SME? Enterprise? Freelancer? ต้องระบุชัดไม่ใช่ "ทุกคน" |
| **Value Propositions** | แก้ปัญหาอะไรที่คู่แข่งทำไม่ได้? (ไม่ใช่แค่ Feature list) |
| **Channels** | ลูกค้ารู้จักเราผ่านทางไหน? (SEO, Ads, Referral, Partner, Community) |
| **Customer Relationships** | Self-service? Onboarding? CSM? Community? |
| **Revenue Streams** | Subscription Monthly/Annual, Freemium, Usage-based, One-time setup |
| **Key Activities** | Dev, Marketing, Customer Success, R&D |
| **Key Resources** | Dev Team, Infra/Cloud, Domain Expertise, Data |
| **Key Partnerships** | Cloud Provider, Payment Gateway, Channel Partners, Integrations |
| **Cost Structure** | Server cost, Dev salary, Marketing CAC, Support |

### หลักการสำคัญ:
- **เริ่มจาก "ทำไม" ก่อน "อะไร"** — อย่าเริ่มที่ Feature แต่เริ่มที่ปัญหาลูกค้า
- **BMC ไม่ใช่เอกสารทำครั้งเดียว** — ต้องกลับมาปรับทุกครั้งที่ Pivot หรือ Scale
- **ใช้สื่อสารในทีม** — BMC เป็นภาษากลางของ Dev, Marketing, Sales

## Framework 2: Lean Canvas — กระดุมเม็ดแรก

> 📖 สกัดจาก: คลิป "Lean Canvas อนาคตธุรกิจตัดสินกันที่กระดุมเม็ดแรก" (Tmvl4dIhlRE)

### 3 Milestones สำหรับ SaaS:

```
[1] Problem-Solution Fit        [2] Product-Market Fit        [3] Business-Market Fit
     "ปัญหามีจริง                    "ตลาดต้องมี                    "โมเดลทำกำไรได้
      วิธีแก้ใช้ได้"                  ลูกค้ายอมจ่าย"                  Scale ได้จริง"
```

### ช่อง Lean Canvas สำหรับ SaaS:

| ช่อง | คำถามสำคัญ |
|---|---|
| **Problem** | ลูกค้า SaaS ของเรากำลังเจ็บปวดเรื่องอะไร? (Top 3 Problems) |
| **Customer Segments** | ใครคือ Early Adopter ที่ยอมจ่ายตั้งแต่ Beta? |
| **Unique Value Proposition** | ประโยคเดียวที่อธิบายว่าทำไมต้องใช้เรา ไม่ใช่คู่แข่ง |
| **Solution** | 3 Features หลักที่แก้ 3 Problems ข้างต้น |
| **Unfair Advantage** | อะไรที่คู่แข่งลอกไม่ได้? (Domain expertise, Data, Network effect) |
| **Revenue Streams** | Pricing model ที่ลูกค้ายอมจ่ายซ้ำทุกเดือน |
| **Cost Structure** | MRR ต้องเท่าไหร่ถึง Break-even? |
| **Key Metrics** | MRR, Churn Rate, LTV, CAC, Activation Rate |
| **Channels** | ช่องทางที่เข้าถึง Early Adopter ได้เร็วที่สุด |

### หลักการสำคัญ:
- **ตรวจสอบสมมติฐาน** — ทุกช่องคือ "สมมติฐาน" ที่ต้องพิสูจน์ ไม่ใช่ข้อเท็จจริง
- **ห้ามข้ามขั้น** — SaaS 42% ล้มเหลวเพราะรีบ build ทั้งที่ยังไม่ผ่าน Problem-Solution Fit
- **Lean Canvas เป็น Hypothesis Log** — ไม่ใช่ Business Plan ที่เขียนแล้ววาง

## Framework 3: Jobs-to-be-Done (JTBD) — ลูกค้า "จ้าง" SaaS มาทำอะไร

> 📖 สกัดจาก: คลิป "สิ่งที่คุณขายไม่สำคัญ วิธีที่คุณขายต่างหากที่สำคัญ" (zI_NCZ7NEBA)

### 3 ประเภท Job ที่ลูกค้า SaaS ต้องการ:

| ประเภท Job | ตัวอย่างในบริบท SaaS |
|---|---|
| **Functional Job** | ลดเวลาทำงานจาก 3 ชม. เหลือ 10 นาที, ลด Human Error |
| **Emotional Job** | รู้สึกมั่นใจว่าข้อมูลปลอดภัย, หายกังวลเรื่อง Compliance |
| **Social Job** | ใช้แล้วดูเป็นมืออาชีพต่อลูกค้า/หัวหน้า, ทีมยอมรับ |

### Switch Moment — ทำไมลูกค้าถึงย้ายมาใช้ SaaS ของเรา:
คือจังหวะที่ลูกค้าตัดสินใจ "ทิ้ง" วิธีเดิม (Excel, Manual, คู่แข่ง) แล้วย้ายมาใช้เรา

### The Four Forces — 4 แรงผลักที่ตัดสินการซื้อ SaaS:

```
แรงผลักให้เปลี่ยน (ช่วยเรา):
  ① Push  — ปัญหาที่ทนไม่ไหวกับระบบเดิม (ช้า, แพง, บั๊กเยอะ)
  ② Pull  — ภาพที่ดีกว่าที่ SaaS ของเราสัญญา (เร็วขึ้น 10x, Dashboard สวย)

แรงต้านไม่ให้เปลี่ยน (ศัตรูเรา):
  ③ Anxiety — "ย้ายข้อมูลยากไหม?" "ถ้าระบบล่มล่ะ?" "ต้องเทรนทีมใหม่?"
  ④ Habit   — "ก็ใช้ Excel มา 10 ปีแล้ว ทำไมต้องเปลี่ยน"
```

### วิธีใช้ Four Forces กับ SaaS:
- **เพิ่ม Push:** เขียน Content แสดงให้เห็นปัญหาที่ลูกค้ายังไม่รู้ตัว
- **เพิ่ม Pull:** Demo ให้เห็นผลลัพธ์ที่ว้าว, Case Study ลูกค้าจริง
- **ลด Anxiety:** Free trial, Money-back guarantee, ช่วย Migration ฟรี
- **ลด Habit:** Onboarding ง่ายมาก, Import ข้อมูลเดิมได้ 1 คลิก

## Framework 4: Design Thinking 5 Steps — คิดก่อน Build

> 📖 สกัดจาก: คลิป "5 Steps of Design Thinking: หาไม่เจออาจล้มเหลว" (bJag8_Txcac)

```
[Empathize] → [Define] → [Ideate] → [Prototype] → [Test]
   ฟังลูกค้า    นิยามปัญหา   ระดมไอเดีย   สร้างต้นแบบ    ทดสอบจริง
```

| ขั้นตอน | ทำอะไรในบริบท SaaS |
|---|---|
| **Empathize** | สัมภาษณ์ลูกค้าเป้าหมาย 5-10 คน, สังเกต Workflow จริง, หา Pain Point ที่ซ่อน |
| **Define** | สรุป "ปัญหาที่แท้จริง" เป็น 1 ประโยค (Point of View Statement) |
| **Ideate** | ระดมสมอง Feature/Solution 10+ แบบ โดยไม่ตัดสินก่อน → คัดเหลือ Top 3 |
| **Prototype** | สร้าง MVP แบบเร็ว (Figma mockup หรือ Landing Page + Waitlist) |
| **Test** | ให้ Early Adopter ลองใช้จริง → เก็บ Feedback → วนซ้ำ (Iterate) |

### หลักการสำคัญ:
- **ไม่ใช่ลำดับตายตัว** — ติดขั้นไหน ย้อนกลับได้เสมอ (Non-linear)
- **"Fail Fast, Fail Cheap"** — ผิดพลาดเร็วในขั้น Prototype ถูกกว่าผิดพลาดตอน Launch
- **เลิกมองลูกค้าเป็นตัวเลข** — ฟังให้เยอะ ถามว่า "ทำไม" ซ้ำๆ

## Framework 5: Pirate Metrics (AARRR) — ระบบวัดผล SaaS Funnel 🆕

> 📖 มาจาก: Dave McClure (500 Startups) — Gold Standard ของ SaaS ทั่วโลก

### ทำไมต้องมี:
Framework 1-4 ช่วยวางแผนและสร้าง แต่ **ขาดระบบวัดผล** ว่า SaaS ทำงานจริงไหม — AARRR เติมตรงนี้

```
[A] Acquisition → [A] Activation → [R] Retention → [R] Revenue → [R] Referral
  ลูกค้ารู้จักเรา    ใช้ครั้งแรก "Aha!"   กลับมาใช้ซ้ำ    ยอมจ่ายเงิน     บอกต่อคนอื่น
```

| Stage | Key Metric สำหรับ SaaS | เป้าหมาย |
|---|---|---|
| **Acquisition** | จำนวน Visitor / Sign-up | ลูกค้าเจอเราผ่านช่องทางที่ถูกต้อง |
| **Activation** | % ที่ถึง "Aha! Moment" | ลูกค้าเห็นคุณค่าหลักภายใน 5 นาทีแรก |
| **Retention** | MAU, GRR ≥ 85% | ลูกค้ากลับมาใช้ซ้ำ ไม่ Churn |
| **Revenue** | MRR, LTV, CAC Payback ≤ 12 เดือน | ทำเงินได้จริงและคุ้มทุน |
| **Referral** | NPS ≥ 40, Viral Coefficient | ลูกค้าบอกต่อโดยไม่ต้องจ่าย Ads |

### เชื่อมกับ Framework อื่น:
- **Lean Canvas** → ช่อง Key Metrics ให้ใช้ AARRR เป็นฐาน
- **JTBD** → ถ้า Activation ต่ำ = ลูกค้ายังไม่เจอ "Job" ที่เราแก้ได้
- **Design Thinking** → ถ้า Retention ต่ำ = ต้องกลับไป Empathize ใหม่

### กฎเหล็ก:
- **Fix Retention ก่อน Scale** — ถ้า GRR < 85% อย่าเพิ่มงบ Marketing
- **Activation คือด่านชี้เป็นชี้ตาย** — ถ้า User ไม่ถึง Aha! Moment ใน Session แรก จะ Churn แน่

## Framework 6: SaaS Pricing Strategy — ตั้งราคายังไงให้ลูกค้ายอมจ่าย 🆕

> 📖 มาจาก: Patrick Campbell (ProfitWell), OpenView Partners — Best Practice 2024-2025

### ทำไมต้องมี:
Framework อื่นพูดเรื่อง Revenue Streams แต่ไม่ลงลึกเรื่อง **วิธีตั้งราคา** ซึ่งเป็นจุดเป็นจุดตายของ SaaS

### Value Metric — หัวใจของการตั้งราคา SaaS
คือหน่วยวัดที่ **สะท้อนคุณค่าจริง** ที่ลูกค้าได้รับ ไม่ใช่แค่จำนวน User

| ❌ Value Metric ที่ไม่ดี | ✅ Value Metric ที่ดี |
|---|---|
| จำนวน User (Per Seat) | จำนวน Invoice ที่สร้าง |
| Storage ที่ใช้ (GB) | จำนวน Transaction ที่ Process |
| จำนวน Project | Revenue ที่ช่วยลูกค้าสร้าง |

### Pricing Model ยอดนิยม (2025):

| Model | เหมาะกับ | ตัวอย่าง |
|---|---|---|
| **Freemium** | SaaS ที่ต้องการ Viral Growth | Slack, Notion, Canva |
| **Flat-Rate** | SaaS ที่ง่ายไม่ซับซ้อน | Basecamp |
| **Tier-Based** | SaaS ที่มีลูกค้าหลายขนาด | HubSpot, Ahrefs |
| **Usage-Based** | SaaS ที่ Value ผันตามปริมาณ | AWS, Twilio, Stripe |
| **Hybrid (61% ใช้)** | SaaS ที่ต้องการทั้ง Predictable + Scalable | Snowflake, Datadog |

```
Hybrid Model = Base Fee (ค่าแรกเข้า/เดือน) + Usage-Based (ตามการใช้งานจริง)
```

### Tier-to-Message Matrix:
สร้างตารางเชื่อม Pricing Tier กับ Value Proposition:

| Tier | กลุ่มเป้าหมาย | Value Proposition | Value Metric |
|---|---|---|---|
| Free | Freelancer / ลองใช้ | ลดเวลา X ได้ฟรี | 10 transactions/เดือน |
| Pro | SME | ระบบครบ + Report | 1,000 transactions/เดือน |
| Enterprise | องค์กรใหญ่ | Security + SLA + API | Unlimited + Dedicated Support |

### เชื่อมกับ Framework อื่น:
- **BMC** → ช่อง Revenue Streams ให้ใช้ Pricing Model ข้างบน
- **JTBD** → Value Metric ต้องตรงกับ Functional Job ที่ลูกค้าจ้างเราทำ
- **Lean Canvas** → Cost Structure vs Revenue ต้อง Balance กัน
- **AARRR** → ราคาต้องไม่เป็นอุปสรรค Activation (ให้ลองฟรีก่อน)

## Framework 7: Switch Interview — เทคนิคสัมภาษณ์แบบ Pro 🆕

> 📖 มาจาก: Ash Maurya (Running Lean) + Bob Moesta (Demand-Side Sales)

### ทำไมต้องมี:
Framework JTBD สอนเรื่อง Four Forces แต่ไม่ได้สอน **วิธีสัมภาษณ์จริง** เพื่อค้นหาข้อมูลเหล่านี้จากลูกค้าตัวจริง

### Switch Interview คืออะไร?
เทคนิคสัมภาษณ์ที่ **ย้อน Timeline** การตัดสินใจของลูกค้า เพื่อหา Four Forces ที่ซ่อนอยู่

```
Timeline ของลูกค้า:
[ใช้ระบบเดิม] → [เริ่มรู้สึกเจ็บปวด] → [เริ่มหาทางออก] → [เจอ SaaS ของเรา] → [ตัดสินใจซื้อ]
                 ←────── ย้อนถามจากตรงนี้ ──────→
```

### 5 คำถาม Switch Interview:

| # | คำถาม | ค้นหา Force ไหน |
|---|---|---|
| 1 | "ก่อนหน้านี้คุณใช้อะไรอยู่?" | Habit (ความเคยชิน) |
| 2 | "จุดไหนที่รู้สึกว่า ทนไม่ไหวแล้ว?" | Push (แรงผลัก) |
| 3 | "ตอนเจอเราครั้งแรก อะไรทำให้สนใจ?" | Pull (แรงดึงดูด) |
| 4 | "มีอะไรที่ลังเลก่อนจะตัดสินใจไหม?" | Anxiety (ความกังวล) |
| 5 | "อะไรทำให้ตัดสินใจเปลี่ยนในที่สุด?" | Switch Moment (จุดเปลี่ยน) |

### Desirability-Viability-Feasibility Test:
ผลจาก Switch Interview ต้องผ่าน 3 ด่าน:

| ด่าน | คำถาม |
|---|---|
| **Desirability** | ลูกค้าต้องการจริงไหม? (Job มีอยู่จริง?) |
| **Viability** | ทำแล้วเราอยู่รอดไหม? (ลูกค้ายอมจ่ายพอ?) |
| **Feasibility** | ทำได้จริงไหม? (ทีมและเทคโนโลยีพร้อม?) |

### เชื่อมกับ Framework อื่น:
- **JTBD Four Forces** → ได้ข้อมูลจริงจากลูกค้า ไม่ใช่เดาเอง
- **Design Thinking Empathize** → Switch Interview เป็นเทคนิค Empathize ระดับ Pro
- **Lean Canvas** → ผลสัมภาษณ์ใช้ Validate ช่อง Problem + UVP
